# SA Toolkit — Onboarding Guide

**Audience**: New DigitalOcean SA joining the AI-assisted operating model.
**Time to complete**: ~30 minutes.

---

## Prerequisites

| Requirement | Check |
|-------------|-------|
| Claude Code CLI installed | `claude --version` |
| Git access to `DO-Solutions/sa-toolkit` | `git ls-remote https://github.com/DO-Solutions/sa-toolkit.git` |
| DO org API key from SA lead | Check `#sa-ai-toolkit` Slack |
| Git configured with your identity | `git config --global user.name` |

**Install Claude Code** (if not already):
```bash
npm install -g @anthropic/claude-code
```

**Set org API key**:
```bash
export ANTHROPIC_API_KEY=<key-from-sa-lead>
echo 'export ANTHROPIC_API_KEY=<key-from-sa-lead>' >> ~/.zshrc
```

**Configure git identity** (if not already):
```bash
git config --global user.name "Your Name"
git config --global user.email "you@digitalocean.com"
```

## Initial setup

### 1. Clone the toolkit

```bash
mkdir -p ~/do
git clone https://github.com/DO-Solutions/sa-toolkit.git ~/do/sa-toolkit
```

### 2. Set the `sa` alias

Add to `~/.zshrc` or `~/.bashrc`:
```bash
alias sa='cd ~/do/sa-toolkit && claude'
```

Reload:
```bash
source ~/.zshrc
```

### 3. Verify

```bash
sa
```

Claude starts, loads `CLAUDE.md`, and checks the current git branch. On `main` it will tell you there's no active engagement and prompt you to run `/engage`. That's correct.

## How the toolkit is structured

```
sa-toolkit/
├── projects/                    ← active customer work (one folder per customer)
│   └── acme-corp/
│       ├── engagement.yaml      ← metadata: stage, fit score, products, SA
│       ├── deliverables/        ← outputs from /commands — clean, committed
│       │   ├── customer-profile.md
│       │   ├── customer-qualification.md
│       │   ├── cost-comparison.md
│       │   └── poc-plan.md
│       └── raw/                 ← GITIGNORED — your working files, logs, configs, PII
├── skills/                      ← reusable patterns extracted from past work
└── knowledge/                   ← completed, anonymized engagement captures
```

**Two types of files in a project:**

- `deliverables/` — generated by `/commands`, clean enough to share internally, committed to git
- `raw/` — anything goes: test scripts, iperf output, customer emails, scratch notes. Never committed.

**Branch per engagement:**

Every customer gets its own branch (`customer/[slug]`). Claude knows which engagement is active by reading the branch name. Main branch = shared toolkit only.

---

## Slash commands

| Command | What it does | Output |
|---------|-------------|--------|
| `/engage` | Initialize new engagement — creates folder, yaml, branch | `projects/[slug]/` |
| `/profile` | Customer one-pager from raw notes | `deliverables/customer-profile.md` |
| `/qualify` | Technical fit score + recommended motion | `deliverables/customer-qualification.md` |
| `/cost` | DO vs incumbent cost comparison | `deliverables/cost-comparison.md` |
| `/poc-init` | POC plan, checklist, runbook | `deliverables/poc-plan.md` |
| `/capture` | Close engagement, write to knowledge base | `knowledge/engagements/[date]-[slug].md` |

## End-to-end workflow

Here's the complete cycle using the WireGuard networking validation as the example.

---

### Step 1: Pull latest before you start

```bash
cd ~/do/sa-toolkit
git pull origin main
```

Always start from a fresh main. Other SAs may have merged new knowledge since your last session.

---

### Step 2: Initialize the engagement

```bash
sa
```

```
/engage
```

Claude asks for customer name, AE, context, stage, and products. You answer in plain text:

```
Customer: Acme Corp
AE: Sarah Chen
Context: Evaluating WireGuard VPN to connect Droplets across NYC and AMS regions
Stage: qualifying
Products: Droplets, VPC, Cloud Firewall
```

Claude creates:
```
projects/acme-corp/
├── engagement.yaml
├── deliverables/
└── raw/
```

And checks out branch `customer/acme-corp`. From this point, every command knows it's working on Acme Corp.

---

### Step 3: Profile the customer

Paste in whatever you have — Slack thread, call notes, CRM export:

```
/profile

[raw notes here]
```

Claude extracts and structures a one-pager, saves it to `deliverables/customer-profile.md`. Review and fill in any `TBD` fields.

---

### Step 4: Qualify

```
/qualify
```

Claude loads the profile for context, asks for any missing scale/budget/timeline info, scores fit, recommends a motion. Output saved to `deliverables/customer-qualification.md`.

Claude also searches `knowledge/engagements/` automatically — if a past SA ran a similar WireGuard test, you'll see it referenced here.

---

### Step 5: Do the technical work

Put all your working files in `raw/` — it's gitignored, so dump anything there freely:

```bash
# Test configs, scripts, iperf output, customer IPs, etc.
~/do/sa-toolkit/projects/acme-corp/raw/
```

Use Claude for reference while you work:

```
Read skills/architectures/ and help me plan a WireGuard hub-and-spoke setup
for 2 regions. Generate a runbook.
```

---

### Step 6: Build the cost comparison

```
/cost

Current: AWS
20x EC2 t3.medium, VPN Gateway ($0.05/hr), 500GB/mo egress
Region: us-east-1
```

Claude outputs a side-by-side table. Review assumptions, adjust as needed. Saved to `deliverables/cost-comparison.md`.

---

### Step 7: POC plan (if proceeding)

```
/poc-init
```

Claude pre-fills from the qualification deliverable — products, risks, success criteria. Output saved to `deliverables/poc-plan.md`.

During the POC, keep all output files in `raw/`. When it's done, you have everything needed to capture.

---

### Step 8: Capture

After the engagement resolves (win, loss, or validated):

```
/capture
```

Claude reads everything — `engagement.yaml`, all deliverables — and asks you only what it can't infer:

- Outcome (closed-won / closed-lost / technical-validation / stalled)
- Key result metric
- What worked / what didn't
- Any pattern worth extracting to `skills/`?

Claude generates:
- `knowledge/engagements/YYYY-MM-DD-acme-corp.md` — anonymized if needed
- `skills/architectures/wireguard-overlay-network.md` — if you found a reusable pattern

---

### Step 9: Commit and open PR

```bash
git add knowledge/ skills/ projects/acme-corp/engagement.yaml
git commit -m "capture: acme-corp — WireGuard validation, 900Mbps same-DC"
git push origin customer/acme-corp
```

Open PR: `customer/acme-corp → main`

SA lead reviews, merges. Branch deleted. The knowledge is now in `main` for every SA.

## Contributing back

| When | What |
|------|------|
| After every POC or significant discovery | `/capture`, commit, PR |
| When you find a reusable architecture pattern | Add to `skills/architectures/` |
| When a competitor comes up | Add to `knowledge/competitive/` |
| When pricing data is stale | Update `skills/cost-models/` with new date |

**Before committing**: ask Claude to review the capture output — *"Check this engagement entry for gaps, inaccuracies, or anything that shouldn't be committed."*

**Anonymization**: Replace customer names with slugs if no reference permission. `Acme Corp` → `mid-market-saas-nyc` is fine for the knowledge entry. The `projects/` folder can keep the real name since it's on a branch, not main.

## Troubleshooting

**Claude doesn't see slash commands**
→ Run `sa` (the alias), not just `claude`. Check: `ls ~/do/sa-toolkit/.claude/commands/`

**Claude isn't loading engagement context**
→ Check your branch: `git branch --show-current`. Should be `customer/[slug]`. If on `main`, run `/engage` or `git checkout customer/[slug]`.

**API key errors**
→ `echo $ANTHROPIC_API_KEY` — if empty, re-export and add to shell profile.

**Git push rejected**
→ Always push a `customer/` branch, never push directly to `main`.

**Accidentally committed raw/ files or PII**
→ Tell the SA lead immediately. Do not attempt to fix with a force-push on your own.
