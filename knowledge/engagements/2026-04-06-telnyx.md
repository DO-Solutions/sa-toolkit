# Engagement: Telnyx — WireGuard LB at Scale on Self-Managed Kubernetes

## Context

Telnyx is an existing DO customer running self-managed Kubernetes via Rancher (RKE2). They are building a WireGuard-based VPN service where each customer gets their own WireGuard pod, and all pods are fronted by a single DigitalOcean Network Load Balancer. They came to us to validate whether the REGIONAL_NETWORK LB can handle this at scale and to understand any platform limitations before going to production.

Primary concern: UDP routing, per-port forwarding, pod-per-customer isolation, and the 40-rule LB ceiling.

## What we did

Three phases of testing across one week:

**Phase 1 (2026-03-25):** Baseline — single WireGuard pod, DOKS, REGIONAL_NETWORK LB on UDP 51820. Validated handshake, data transfer, MTU, and session persistence.

**Phase 2 (2026-03-26):** Architecture options — tested 4 topology variants with multiple pods on DOKS. Identified that only two are viable: separate LB per pod (expensive) and shared gateway pod (cheapest). Identified why single-service multi-port and source-IP affinity don't work (K8s routing and WireGuard crypto constraints respectively).

**Phase 3 (2026-03-30):** Scale test — 39 independent WireGuard pods, unique UDP ports (32000–32038), single REGIONAL_NETWORK LB, self-managed RKE2 cluster (4 nodes). Hit and confirmed the 40-rule ceiling. Separately validated "port 0" (any-to-any UDP) which removes the ceiling.

## Results

- **39 pods on one LB: confirmed.** All 8 test categories pass: handshake, simultaneous tunnels, cross-node routing, pod restart, rule limit, resource usage, rule management, source IP behavior.
- **Port 0 (any-to-any UDP): confirmed.** 5 pods on arbitrary ports never added as LB rules — all pass. Single rule replaces per-port management entirely.
- **Resource footprint is negligible:** ~1m CPU, ~4 MiB RAM per WireGuard pod. Node resources are not the bottleneck.
- **LB rule propagation:** ~19s for a new rule to take effect. Account for this in provisioning flows.
- **Pod restart:** Replacement in ~5s with key persistence via K8s Secrets. Other tunnels unaffected.
- **Cost at scale:** $12/mo per LB regardless of customer count (with port 0). ~$0.31/customer/month at 39 customers per LB.

## What worked

- DO REGIONAL_NETWORK LB handles WireGuard UDP correctly out of the box: session persistence, correct per-port routing, health check integration.
- Port 0 (any-to-any UDP) is a clean solution to the rule ceiling — single LB rule, no per-customer rule management.
- Storing server private keys in K8s Secrets solves pod restart key persistence completely.
- kube-proxy NodePort routing works correctly even when pods are spread across nodes — NLB only needs to hit any backend, kube-proxy does the rest.
- `PersistentKeepalive = 25` reliably prevents UDP session timeouts on the LB.

## What didn't work (gotchas)

**1. NLB local route required on self-managed K8s nodes — nothing works without it.**

DO REGIONAL_NETWORK LBs use Direct Server Return (DSR). Packets arrive at nodes with the NLB IP as the destination. Without a local route, the kernel drops them silently. On DOKS, the CCM adds this automatically. On self-managed K8s (Rancher/RKE2), you must add it manually and persist via systemd:

```bash
ip route add to local <NLB_IP> dev eth0
```

This was the biggest operational gotcha. Any SA setting up REGIONAL_NETWORK LB with self-managed K8s must communicate this upfront.

**2. Health check DaemonSet with `hostNetwork: true` required.**

NodePort-based health checks don't work with REGIONAL_NETWORK LBs. The health check response comes from the pod's node, not the node the NLB checked, so the NLB marks nodes as unhealthy. The fix is a DaemonSet with `hostNetwork: true` on a fixed port — ensures every node responds directly to health checks.

**3. Source IP is NATed by kube-proxy.**

WireGuard pods see a kube-proxy internal IP (`10.42.x.x`) as the client endpoint, not the real client public IP. This is a K8s behavior (MASQUERADE on cross-node forwarding). Telnyx should use the WireGuard public key for customer identification rather than source IP. If real source IP is critical, `externalTrafficPolicy: Local` works but requires pods to run on the same node the NLB hits (hurts distribution).

**4. 40-rule ceiling.**

Hit at 39 customers per LB (39 UDP + 1 TCP health check = 40). Resolved by port 0 feature, but SAs should be aware this limit exists when customers ask about per-port routing models before port 0 is available.

## For the next SA who sees this customer type

**Telecom / VPN-as-a-service customers running self-managed K8s who want per-customer WireGuard tunnels through a single LB:**

The architecture works. Use port 0 (any-to-any UDP) on a REGIONAL_NETWORK LB — one LB rule, unlimited pods, $12/mo. The two mandatory gotchas are the NLB local route and the hostNetwork health check DaemonSet; get these right upfront or nothing will work. Skip source-IP-based logging and use WireGuard public keys for customer identification. Phase 2 test results are useful for explaining to customers why "just put them all on one Service" and "use source-IP affinity" don't work — both are reasonable questions that have non-obvious answers.
